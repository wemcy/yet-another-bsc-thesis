\section{Megvalósítás}\label{ch:implementation}

\subsection{Fejlesztési környezet és eszközök}

Fejlesztői környezetnek több szoftvert és eszközt is használtam. A backend fejlesztéséhez Visual Studiot\footnote{\url{https://visualstudio.microsoft.com/}. Utolsó elérés: 2026.04.22.} használtam, a frontendhez Visual Studio Code-ot\footnote{\url{https://code.visualstudio.com/}. Utolsó elérés: 2026.04.22.} használtam. Ezek a fejlesztői környezetek könnyedén hozzáférést biztosítanak a szükséges bővítményekhez, mint például az Entity Framework Core eszközei, a Vue támogatás, valamint a Docker integráció.

A fejlesztéshez szükséges a már korábban említett Docker, Docker Compose, ami a konténerizációval a rendszer futtatásához szükséges.

Verziókezelés Git\footnote{\url{https://git-scm.com/}. Utolsó elérés: 2026.04.22.}-en keresztül történik, a repository pedig a GitHubon\footnote{\url{https://github.com/}. Utolsó elérés: 2026.04.22.} található.

A backendben a függőségek NuGet csomagkezelővel vannak kezelve. ASP.NET Core 10-re épül a projekt, és olyan csomagokat használ, mint az Entity Framework Core, valamint a Microsoft Identity platformhoz kapcsolódó csomagok.

Frontend oldalon a Vite és a Vue 3 használata mellett TypeScript is jelen van, ami a típusbiztonságot és a fejlesztési élményt javítja. A csomagok pnpm-en keresztül vannak kezelve, és a package.json fájlban vannak felsorolva. A projekt egyéb részén is pnpm van használva, például a frontend build és a tesztelési parancsoknál.

E2E tesztelés Cypress\cite{cypress} segítségével történik, amely a frontend és backend integrációját teszteli. 

A REST API specifikáció alapján OpenAPI CLI Generator\footnote{\url{https://openapi-generator.tech/}. Utolsó elérés: 2026.04.22.} generál egy frontend API klienst, és egy backend projektet, hogy a fejlesztés során a backend és a frontend szinkronban legyenek. Ehhez szükséges Java 17 vagy annak újabb verziója.

Az adatbázis megtekintéséhez Adminer\footnote{\url{https://www.adminer.org/}. Utolsó elérés: 2026.04.22.} van bekötve a konténeres környezetbe, amely egy könnyen használható webes adatbázis-kezelő eszköz.

A könnyű fejlesztés érdekében a legtöbb gyakran használt parancs, mint például a build\footnote{ forráskód gépi nyelvre való átalakítása}, tesztelés vagy a konténeres futtatás, PowerShell scriptek segítségével van automatizálva.

\subsubsection{Modell réteg}\label{sec:adatbazis}

Az Ingredient entitás az összetevőket reprezentálja, amelyek a receptekhez kapcsolódnak.

\subsubsection{Keresés és szűrés}

Az egyik legfontosabb tényező a fejlesztés során azt volt, hogy a keresés milliszekundumok alatt történjen meg. Az allergének emiatt bitflagként vannak tárolva. Ennek köszönhetően óriási teljesítménybeli előnyre lehet szert tenni, mintha minden allergént külön szöveges mezőként tárolnánk. Akár több millió recept között is gyorsan meg lehet találni azokat, amelyek az adott allergén szűrőnek megfelelnek. A szűrés implementációja a~\ref{lst:recipe-filter} kódrészletben látható.

\begin{lstlisting}[style=csharp, caption={Receptszűrés megvalósítása}, label={lst:recipe-filter}]
public IQueryable<Recipe> ApplyFilters(IQueryable<Recipe> query)
{
    return query
        .ConditionalWhere(HasIncludeFilter, x => (x.Allergens & IncludeAllergens) != AllergenType.None)
        .ConditionalWhere(HasExcludeFilter, x => (x.Allergens & ExcludeAllergens) == AllergenType.None);
}
\end{lstlisting}

A kódban is jól látható, hogy a szűrést a bit műveletek végzik el. Ennek köszönhetően a kód gyorsan hajtódik végre, még nagy adatmennyiség esetén is. Ezt még az adatbázis lekérdezés struktúrájával is tovább lehet optimalizálni azzal, hogy elsőként csak a szűrés feltételeit hajtjuk végre. Ez a megközelítés lehetővé teszi, hogy a szűrés SQL szinten kerüljön végrehajtásra, elkerülve a felesleges adatok memóriába töltését. Ez a megközelítés lehetővé teszi, hogy egyetlen egész szám reprezentálja az összes allergént, ami jelentősen csökkenti a lekérdezések komplexitását és növeli a teljesítményt.

A keresések megvalósításához indexek vannak használva az adatbázisban, amelyek lehetővé teszik a gyors keresést. Emellett a rendszer vektor alapú keresést is alkalmaz. Az egyes rekordokhoz tartozó szöveges tartalmak numerikus reprezentációvá (vektorokká) kerülnek átalakításra. Ezeken a vektormezőkön speciális indexek kerülnek kialakításra, amelyek lehetővé teszik a hasonlóság alapú keresést.
Ennek köszönhetően a keresési műveletek nagy adatmennyiség esetén is hatékonyan hajthatóak végre.

\subsection{Frontend}

\subsubsection{State management}

A frontend állapotkezeléséhez Pinia\footnote{\url{https://pinia.vuejs.org/}. Utolsó elérés: 2026.04.22.} van használva. Ez a Vue 3-hoz készült könyvtár, amely lehetővé teszi a globális állapot kezelését a különböző komponensek között. A Pinia használata megkönnyíti az adatok megosztását és a komponensek közötti kommunikációt. Ennek köszönhetően a kommunikációs logika egy helyen van kezelve, könnyedén módosítható.

\subsubsection{Komponensek}

A frontend két fajta komponensből épül fel: a nézetkomponensek, amelyek az egyes oldalak megjelenítéséért felelősek, valamint az általános komponensek, amelyek a navigációs sávot, láblécet és egyéb újrafelhasználható elemeket tartalmazzák. Ez a megközelítés lehetővé teszi a kód újrafelhasználhatóságát és karbantarthatóságát.

Pár kiemelendő komponens a következő:
\begin{itemize}
    \item \texttt{RecipeCard} — egy recept kisebb nézetét jeleníti meg, annak képét, nevét és felhasználóját. Ez a komponens az oldal sok pontján megjelenik, például a kezdőlapon, vagy a recept listázó oldalakon.
    \item \texttt{RecipeForm} — egy recept létrehozásáért és szerkesztéséért felelős komponens, amely egy űrlapot jelenít meg a recept adataival. A frontendhez tartozó logika nagy része ebben a komponensben van megvalósítva.
    \item \texttt{ProfileCard} — egy felhasználó képét és nevét jeleníti meg. Ez a komponens a kezdőlapon és a profil oldalakon jelenik meg.
    \item \texttt{IngredientList} — egy összetevő listát jelenít meg, amely a recepthez tartozó összetevőket mutatja. Ez a komponens a recept részletező oldalán jelenik meg.
    \item \texttt{CommentSection} — egy komment szekciót jelenít meg, amely a recepthez tartozó kommenteket mutatja. 
\end{itemize}

Ezek mellett is számos kisebb-nagyobb komponens van, ami jól szemlélteti, mennyire
fontos a komponens alapú megközelítés a kód újrafelhasználhatósága és karbantarthatósága szempontjából.

\subsubsection{Routing}

A frontend routingját a Vue Router\footnote{\url{https://router.vuejs.org/}. Utolsó elérés: 2026.04.22.} könyvtár kezeli. Ez lehetővé teszi az egyes oldalak közötti navigációt, valamint a dinamikus útvonalak kezelését. A router konfigurációja meghatározza, hogy melyik nézetkomponens jelenjen meg az adott útvonalon. Emellett a router segítségével megvalósítható a jogosultság alapú útvonalvédelem is, amely biztosítja, hogy csak a megfelelő jogosultságokkal rendelkező felhasználók férjenek hozzá bizonyos oldalakhoz.

Az oldalon a következő útvonalak vannak megvalósítva:

\begin{itemize}
	\item \texttt{/} — kezdőlap; komponens: \texttt{HomeView}
	\item \texttt{/profile} — saját profil; komponens: \texttt{ProfileView}; meta: \texttt{requiresAuth=true}
	\item \texttt{/profile/:id} — publikus profil; komponens: \texttt{PublicProfileView}
	\item \texttt{/recipe/:id} — recept részletei; komponens: \texttt{RecipeView}
	\item \texttt{/new-recipe} — új recept létrehozása; komponens: \texttt{NewRecipeView}; meta: \texttt{requiresAuth=true}
	\item \texttt{/recipes} — összes recept listája; komponens: \texttt{AllRecipesView}
	\item \texttt{/edit/:id} — recept szerkesztése; komponens: \texttt{EditRecipeView}; meta: \texttt{requiresAuth=true}
	\item \texttt{/admin/ingredients} — adminisztratív összetevőkezelés; komponens: \texttt{AdminIngredientsView}; meta: \texttt{requiresAuth=true, requiresAdmin=true}
	\item \texttt{/login} — bejelentkezés; komponens: \texttt{LoginView}
	\item \texttt{/:pathMatch(.*)*} — 404 (NotFound); komponens: \texttt{NotFoundView}
\end{itemize}

A jogosultságok ellenőrzése metaadatok alapján történik. A felhasználó hitelesítéséhez szükséges információk egy session cookie formájában kerülnek tárolásra, amely a szerveroldali munkamenet azonosítására szolgál.

\subsection{Fejlesztés közben felmerült problémák}

Fejlesztés közben több olyan probléma is előrkerült, amelyek megoldása nem volt triviális.
Ezek közül néhány:
\begin{itemize}
    \item A backendben használt LINQ lekérdező nyelv segítségével SQL lekérdezések írhatóak funkcionális stílusban. Ehhez több olyan extenstion függvény is készült, amelyek generikus típus-behelyettesítést használnak. Ezek megírása és végiggondolása nem volt egyszerű, de a végeredmény egy könnyen használható és újrahasználható kódrészlet lett, amely jelentősen csökkenti a kódismétlést és javítja a kód olvashatóságát.
    \item Az alkalmazás írásának megkezdéséhez az egész konténerizációs környezet beállítására és a fejlesztési workflow kialakítására is szükség volt. Ez magában foglal több olyan elemet, amivel még soha sem kellett foglalkoznom, mint például a Docker Compose.
    \item CORS\footnote{Cross-Origin Resource Sharing} és autentikációs problémák is felmerültek a frontend és a backend közötti kommunikáció során, különösen a fejlesztési környezetben, ahol a két komponens különböző portokon fut.
    \item A fejlesztés során előfordultak kisebb hibák és inkompatibilitások a különböző könyvtárak és eszközök verziói között, amelyeket folyamatosan kezelni kellett.
\end{itemize}

\subsection{Hibakezelés}

Ha hiba történik, a frontend felülete egyértelműen jelzi ezt a felhasználó számára. Ennek formája lehet egy felugró ablak vagy egy üzenet a képernyőn, amely leírja a hibát és esetleg javaslatot tesz a megoldásra. A backendben a hibák megfelelő HTTP státuszkódokkal vannak visszaadva, és a hibaüzenetek is informatívak, hogy a frontend könnyen kezelni tudja azokat.

Backenden több saját kivételosztály is segíti a hibák kezelését. Ezek különböző logikai hibák jelzésére szolgálnak, amelyek nem a programozási nyelv által dobott kivételek, hanem a rendszer logikájából adódó hibák. Például egy \texttt{EntityNotFoundException} jelzi, hogy egy adott erőforrás nem található, míg egy \texttt{UnauthorizedAccessException} jelzi, hogy a felhasználónak nincs megfelelő jogosultsága egy művelet végrehajtásához. Ezen kivételek kezelése applikáció szintjén történik, ahol a megfelelő HTTP státuszkódokkal és hibaüzenetekkel válaszol a kliensnek. Ez a megközelítés lehetővé teszi, hogy a hibák kezelése egységes és könnyen karbantartható legyen a teljes alkalmazásban.


\section{DevOps}\label{sec:devops}

A DevOps megoldások célja a rendszer egyszerű indíthatóságának és reprodukálhatóságának biztosítása. Az alkalmazás több olyan eszközt és megoldást használ amelyek megkönnyítik a fejlesztést.

\subsubsection{Docker és konténeres futtatás}

Docker használata lehetővé teszi, hogy a fejlesztési és éles környezetek között konzisztens legyen a futtatási környezet. A Docker Compose segítségével könnyedén definiálható és futtatható a teljes alkalmazás, beleértve a backend szolgáltatást, a frontend alkalmazást, valamint az adatbázist és az Adminert is.

Emellett a szoftver úgynevezett volume-kat használ, amelyek lehetővé teszik, hogy két indítás között ne vesszen el adat. Ezekben vannak tárolva az adatbázis fájlok, illetve a képek.

\subsubsection{Nginx}

Az Nginx egy könnyű és nagy teljesítményű webszerver, amelyet a frontend alkalmazás kiszolgálására használunk. Az Nginx konfigurációja lehetővé teszi a statikus fájlok gyors kiszolgálását, valamint a backend API-k felé történő proxyzást, így a frontend és backend közötti kommunikáció zökkenőmentes.

\subsection{CI/CD}

A projekt fejlesztéséhez GitHub repository van használva, amelyben a GitHub Actions segítségével egy CI/CD\footnote{Continuous Integration/Continuous Delivery/Deployment} pipeline került kialakításra. Ez a pipeline automatikusan lefut minden egyes merge művelet során. Sikeres tesztelés után egy zip fájlt állít elő, amely tartalmazza a rendszer futtatásához szükséges fájlokat. Emellett minden komponensből külön Docker image készül. Ezek a GitHub Packages szolgáltatásban kerülnek tárolásra, és a Docker Compose konfiguráció ezekre hivatkozva teszi lehetővé a rendszer futtatását.
