\section{Megvalósítás}\label{ch:implementation}

\subsection{Fejlesztési környezet és eszközök}

Fejlesztői környezetnek több szoftvert és eszközt is használtam. A backend fejlesztéséhez Visual Studiot használtam, frontendhez Visual Studio Code-ot. Ezek a szoftverek könnyedén hozzáférést biztosítanak a szükséges bővítményekhez, mint például az Entity Framework Core eszközei, a Vue támogatás, valamint a Docker integráció.

A fejlesztéshez szükséges a már korábban említett Docker, Docker Compose, ami a konténerizációval a rendszer futtatásához szükséges.

Verziókezéléshez Git-en keresztül történik, a repository pedig a GitHubon található.

A backendben a függőségek NuGet csomagkezelővel vannak kezelve. ASP.NET Core 10-re épül a projekt, és olyan csomagokat használ, mint az Entity Framework Core, valamint a Microsoft Identity platformhoz kapcsolódó csomagok.

Frontend oldalon a Vite és a Vue 3 használata mellett TypeScript is jelen van, ami a típusbiztonságot és a fejlesztési élményt javítja. A csomagok pnpm-en keresztül vannak kezelve, és a package.json fájlban vannak felsorolva. A project egyéb részén is pnpm van használva, például a frontend build és tesztelési parancsoknál.

E2E teszteléshez Cypress-t történik, amely a frontend és backend integrációját teszteli. 

A REST API specifikáció alapján OpenAPI Generator CLI generál egy frontend API klienst, és egy backend projectet, hogy a fejlesztés során a backend és frontend szinkronban legyenek. Ehhez szükséges Java 17 vagy annak újabb verziója.

Az adatbázis megtekintéséhez Adminer van bekötve a konténeres környezetbe, amely egy könnyen használható webes adatbázis-kezelő eszköz.

A könnyű fejlesztés érdekében a legtöbb gyakran használt parancs, mint például a build,tesztelés vagy a konténeres futtatás, PowerShell szkriptek segítségével van automatizálva.
\subsection{Backend}

Az ASP.NET Core alapú rendszer erősen kihasználja a függőségi injektálást. Lehetővé teszi, hogy ne kelljen manuálisan példányosítani a szolgáltatásokat, hanem a keretrendszer automatikusan kezelei azokat. Emellett könnyen kicserélhetővé teszi a különböző implementációkat, például a repository rétegben, ahol az adatbázis műveletek történnek. \subsubsection{Keresés és szűrés}

Egyik legfontosabb tényező fejlesztés során azt volt, hogy a keresés milliszekundumok alatt történjen meg. Az allergének emiatt bitflagként vannak tárolva. Ennek köszönhetően óriási teljesítménybeli előnyre lehet szert tenni. Akár több millió recept között is gyorsan meg lehet találni azokat, amelyek az adott allergén szűrőnek megfelelnek. A szűrés implementációja a~\ref{lst:recipe-filter} kódrészletben látható.

\begin{lstlisting}[style=csharp, caption={Receptszűrés megvalósítása}, label={lst:recipe-filter}]
public IQueryable<Recipe> ApplyFilters(IQueryable<Recipe> query)
{
    return query
        .ConditionalWhere(HasIncludeFilter, x => (x.Allergens & IncludeAllergens) != AllergenType.None)
        .ConditionalWhere(HasExcludeFilter, x => (x.Allergens & ExcludeAllergens) == AllergenType.None);
}
\end{lstlisting}

A kódban is jól látható, hogy a szűrést a bit műveleteket végzik el. Ennek köszönhetően a kód gyorsan hajtódik végre, még nagy adatmennyiség esetén is. Ezt még az adatbázis lekérdezés struktúrájával is tovább lehet optimalizálni azzal, hogy elsőként csak a szűrés feltételeit hajtjuk végre. Ez a megközelítés lehetővé teszi, hogy a szűrés SQL szinten kerüljön végrehajtásra, elkerülve a felesleges adatok memóriába töltését.

Kereséseket megvalósításhoz indexek vannak használva az adatbázisban. Ezek a beállítások alapján előre generál olyan szöveg indexeket, amelyek alapján később gyorsan lehet keresni.

\subsection{Fejlesztés közben felmerült problémák}

Fejlesztés közben több olyan probléma is előrkerült, amelyek megoldása nem volt triviális.
Ezek közül néhány:
\begin{itemize}
    \item A backend használt LinQ nevezetű kiegészítő segítségévél SQL lekérdezeséket lehet írni funkcionális stílusban. Ehhez több olyan extenstion függvény is készült, amelyek generikus típus behelyettesítést  használnak. Ezek megírása és végig gondolása nem volt egyszerű, de a végeredmény egy könnyen használható és újrahasználható kódrészlet lett, amely jelentősen csökkenti a kódismétlést és javítja a kód olvashatóságát.
    \item Az alkalmazás írásának megkezdéséhez, az egész konténerizációs környezet beállítása és a fejlesztési workflow kialakítása szükség volt. Ez magában foglal több olyan elemet, amivel még soha sem kellet foglalkozni, mint például Docker Copose, vagy a korábban említett GitHub Actions, melyek használta nagy nahézséget okozott.
    \item CORS és autentikációs problémák is felmerültek a frontend és backend közötti kommunikáció során, különösen a fejlesztési környezetben, ahol a két komponens különböző portokon futott.
    \item A fejlesztés során előfordultak kisebb hibák és inkompatibilitások a különböző könyvtárak és eszközök verziói között, amelyeket folyamatosan kezelni kellett.
\end{itemize}

\section{DevOps}\label{sec:devops}

A DevOps megoldások célja a rendszer egyszerű indíthatóságának és reprodukálhatóságának biztosítása. Az alkalmaz több olyan eszközt és megoldást használ amelyek megkönnyítik a fejlesztést.

\subsubsection{Docker és konténeres futtatás}

Docker használata lehetővé teszi, hogy a fejlesztési és éles környezetek között konzisztens legyen a futtatási környezet. A Docker Compose segítségével könnyedén definiálható és futtatható a teljes alkalmazás, beleértve a backend szolgáltatást, a frontend alkalmazást, valamint az adatbázist és az Adminer-t is.

Emellett a szoftver ún.\ volume-kat használ, amelyek lehetővé teszik, hogy két indítás között ne vesszen el adat. Ezekben vannak tárolva az adatbázis fájlok illetve képek.

\subsubsection{NgInx}

Az Nginx egy könnyű és nagy teljesítményű web szerver, amelyet a frontend alkalmazás kiszolgálására használunk. Az Nginx konfigurációja lehetővé teszi a statikus fájlok gyors kiszolgálását, valamint a backend API-k felé történő proxyzást, így a frontend és backend közötti kommunikáció zökkenőmentes.

\subsection{CI/CD}

Project feljesztéséhez github repository van használva, amelyben a GitHub Actions segítségével egy CI/CD pipeline van definiálva. Ez a pipeline automatikusan lefut minden merge esetén. Sikeres tesztelés után előállít egy zip fájlt, ami tartalmazza azokat a fájlokat amit szükségesek a rendzser futtatásához. Emellett minden egyes eleméből keszít egy docker image-t. Ezek GitHub Packages-ben vannak tárolva, és a Docker Compose fájlban ezekre hivatkozva lehet futtatni a rendszert.
