\section{Tervezés}\label{sec:design}

\subsection{Feladat részletes leírása}

Az alkalmazás két fő részből áll: egy ASP.NET Core alapú backend szolgáltatásból, és egy Vue és TypeScript alapú frontend egyoldalas alkalmazásból. Fontos, hogy ezen komponensek önállóan is működőképesek legyenek, de együtt egy teljes funkcionalitást nyújtó rendszert alkossanak. Ennek megvalósítása során a stabil API-kapcsolat, a jól definiált adatmodell, valamint a felhasználói élményre fókuszáló frontend kialakítása a fő célok közé tartozik. 

\subsection{Funkcionális követelmények}\label{ch:requirements}

Funkcionális követelmények tekintetében a projekt fő területei a következőek:

\begin{itemize}
	\item felhasználókezelés: regisztráció, bejelentkezés, profil megjelenítése és módosítása
	\item receptkezelés: új recept létrehozása, szerkesztése, törlése, részletes receptnézet
	\item keresés és szűrés: receptlista böngészése, kulcsszavas keresés, szűrés allergének alapján
	\item közösségi funkciók: receptek értékelése, kommentek rögzítése és törlése
\end{itemize}

\subsection{Nem funkcionális követelmények}

A nem funkcionális követelmények a következőek:

\begin{itemize}
	\item átlátható és modern böngészőkben megfelelő megjelenés
	\item jól dokumentált, stabilan használható HTTP API
	\item olyan adatmodell, amely a recept-, komment- és felhasználói adatok közötti kapcsolatokat konzisztensen kezeli
	\item könnyen továbbfejleszthető frontend és backend oldalon
	\item fejlesztői munkát segítő konténeres indítási és build környezet
	\item a rendszer helyi környezetben való könnyű indíthatósága, minimális külső függőséggel
\end{itemize}

\subsection{Architekturális áttekintés}

A projekt réteges, webalkalmazás-jellegű architektúrára épül. A felhasználó egy böngészőben futó kliensalkalmazással lép kapcsolatba, amely HTTP hívásokon keresztül éri el a backend szolgáltatást. A backend végzi az üzleti logika, az adatelérés, a hitelesítéshez kapcsolódó folyamatok, valamint az API-dokumentáció kiszolgálásának jelentős részét. Az adatok perzisztálását egy relációs adatbázis biztosítja.

\begin{figure}[H]
	\centering
	\includegraphics[width=0.8\textwidth]{architecture.png}
	\caption{A rendszer architektúrája}\label{fig:architecture-diagram}
\end{figure}

A fő komponensek:

\begin{itemize}
	\item Frontend: Vue 3 alapú egyoldalas alkalmazás, amely a felhasználói folyamatokat és az oldalak közti navigációt kezeli
	\item API kliens: OpenAPI alapján generálható klienscsomag, amely a frontend és a backend közötti szerződést teszi követhetővé
	\item Backend: ASP.NET Core Web API, amely kontroller-, szolgáltatás- és repository-szintű felelősségekre tagolódik
	\item Adatbázis: PostgreSQL adatbázis, amely a receptek, kommentek, felhasználói adatok és egyéb kapcsolódó információk tárolásáért felelős
	\item Proxy: Nginx alapú fordított proxy, amely a frontend és a backend szolgáltatásokat egyetlen végpont alá helyezi
	\item Docker volumes: az adatbázis adatai és a képek egy-egy megosztott kötetben vannak tárolva 
\end{itemize}

\subsection{Backend architekturális döntések}

A backend architektúra réteges felépítésű. A kontroller réteg felelős a HTTP kérések fogadásáért, a válaszok visszaküldéséért, valamint a bemeneti adatok validálásáért. A szolgáltatásréteg tartalmazza az üzleti logikát, és a kontroller által meghívott műveleteket hajtja végre. A repository réteg az adatbázis-műveletekért felel, elválasztva az adatkezelést az üzleti logikától. Az adatok egy entitásmodell segítségével kerülnek reprezentálásra, amelyek relációs adatbázisban vannak tárolva. Ez a megközelítés elősegíti a kód tisztaságát, a felelősségek elkülönítését és a rendszer karbantarthatóságát.

\subsection{Frontend architekturális döntések}

A frontend architektúra komponensalapú felépítést követ. A Vue 3 keretrendszer használata lehetővé teszi az újra felhasználható, paraméterezhető komponensek létrehozását. A felület nézetkomponensekre van bontva, amelyek az egyes oldalak megjelenítéséért felelősek. A közös elemek, mint például a navigációs sáv és a lábléc, külön komponensként vannak definiálva. Az állapotkezelést egy központi store (Pinia) biztosítja, amely lehetővé teszi az adatok megosztását a különböző komponensek között. A frontend a backenddel REST API-n keresztül kommunikál. Ez a megközelítés biztosítja a frontend és a backend közötti laza csatolást, valamint megkönnyíti a fejlesztést és a tesztelést.

\subsection{REST API tervezés}

Az alkalmazás tervezésénél fontos szempont volt, hogy a backend és a frontend önmagukban is teljes értékű, jól működő komponensek legyenek. Ennek eléréséhez elengedhetetlen, hogy a két rendszer jól dokumentált legyen és következetes kommunikációt folytasson egymással. Ez lehetővé teszi a frontend és a backend párhuzamos fejlesztését, valamint azt, hogy a komponensek külön-külön is könnyen tesztelhetők legyenek.

Erre a problémára a Representational State Transfer (REST)\cite{rest} API tervezési megközelítés nyújt megoldást, amely egy széles körben alkalmazott szabvány a webes szolgáltatások kialakításában. A REST API-k használata biztosítja a frontend és a backend közötti egyszerű, egységes és hatékony kommunikációt.

A megközelítés alkalmazása számos előnnyel jár: támogatja a rendszer skálázhatóságát, megkönnyíti a tesztelést, valamint lehetővé teszi a komponensek egymástól független fejlesztését és karbantartását. A backend szolgáltatás nem függ közvetlenül a frontend implementációjától, így a két réteg lazán csatolt módon működik együtt.

További előny, hogy a rendszer könnyen kiterjeszthető más platformokra is. Amennyiben a jövőben mobilalkalmazás vagy egyéb kliens kerül bevezetésre, ezek ugyanazt az API-t használhatják, így elegendő csupán a megjelenítési réteget az adott platformhoz igazítani.

Ugyanakkor a REST API alkalmazása hátrányokkal is jár. Az API-k tervezése időigényes lehet, különösen összetett üzleti logika esetén. Emellett növeli a komplexitást, mivel több eszközt kell használni fejlesztéshez és több programozási nyelvet kell ismerni.

\subsection{Adatbázis}

Az adatbázis tervezése során a fő szempont az volt, hogy minél kevesebb interakció történjen az adatbázissal natív SQL nyelven írt lekérdezésekkel.  Ennek megoldására az Entity Framework (továbbiakban EF)\cite{ef} keretrendszert használja a backend. Az EF egy objektum-relációs leképező (ORM) eszköz, amely lehetővé teszi, hogy a fejlesztők C\# osztályokkal dolgozzanak, miközben az adatok relációs adatbázisban tárolódnak. Ennek köszönhetően az adatok kapcsolatait és a lekérdezéseket is a C\# kódon keresztül lehet kezelni, anélkül, hogy közvetlenül SQL-t kellene írni. Ennek köszönhetően az adatbázis struktúrája típusos, jól karbantartható. A kapcsolatokat a nyelv megszokott kulcsszavaival lehet kezelni. A rendszer első indításakor üres állapotban, receptek és összetevők nélkül indul el.

\begin{figure}[H]
	\centering
	\includegraphics[width=0.7\textwidth]{umls/databasecontext.jpg}
	\caption{Adatbázis kontextus UML diagramja}\label{fig:database-context-diagram}
\end{figure}

A fenti ábrán látható a DatabaseContext osztály UML diagramja, amely az adatbázisban tárolt entitásokat írja le. Az osztályban definiált DbSet-ek reprezentálják a különböző táblákat az adatbázisban. Ezek mellett helyet kapott néhány függvény, amely minden egyes entitás módosításánál vagy létrehozásánál meghívásra kerül, így biztosítva az adatok konzisztenciáját. Az adatbázis model részletes leírása a \ref{sec:adatbazis} fejezetben található meg.
